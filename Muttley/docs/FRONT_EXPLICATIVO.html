Documentacao do Front - Projeto Muttley

Este material foi feito em duas partes: primeiro uma explicacao didatica para quem nao domina HTML, e depois uma explicacao tecnica para o time de backend.

Parte 1 - Explicacao didatica (HTML basico)

1) O que foi feito no front

Foram implementadas duas telas principais:

  • Tela inicial em src/main/resources/static/index.html
  • Tela de novo cadastro em src/main/resources/templates/cadastro_aluno.html

O estilo visual das duas telas foi centralizado em: src/main/resources/static/css/front/muttley-front.css

2) Estrutura simples de cada tela

Tela inicial

Tem os campos Nome e CPF.

Botoes: Buscar, Importar Lista, Novo Cadastro, Novo Evento.

O CPF tem mascara 000.000.000-00.

Tela novo cadastro

Tem os campos Nome, CPF, Curso, Email.

Botao principal: Salvar.

Tambem usa mascara de CPF.

3) O que significam as partes do HTML

  • <form>: bloco que envia dados para o backend.
  • action="...": para onde os dados vao.
  • method="get|post": tipo de envio.
  • <input>: campo para digitar dados.
  • name="cpf": nome da variavel que o backend vai receber.
  • <button type="submit">: envia o formulario.

4) Exemplo simples da tela inicial

<form action="/alunos/buscar" method="get">
  <input id="cpfBusca" name="cpf" required>
  <button type="submit">Buscar</button>
</form>

Em palavras simples: quando o usuario clica em Buscar, o navegador chama a rota /alunos/buscar enviando o CPF.

5) Mascara de CPF (front)

Foi adicionado JavaScript para formatar o CPF enquanto o usuario digita. Exemplo: 12345678901 vira 123.456.789-01.

A mascara melhora a experiencia do usuario, mas o backend continua validando os dados.
Parte 2 - Explicacao tecnica para devs backend

6) Como o front conversa com o backend

O projeto usa Spring Boot MVC com mix de pagina estatica + template Thymeleaf.

Fluxo A - Home: / entrega static/index.html.
Fluxo B - Ir para cadastro: link para /alunos/novo, controller retorna cadastro_aluno.
Fluxo C - Salvar: form do cadastro envia POST /alunos/salvar com bind em @ModelAttribute Aluno.
Fluxo D - Buscar: form da home envia GET /alunos/buscar?cpf=....

7) Objetos e variaveis que entram no bind

No template de cadastro:

<form th:action="@{/alunos/salvar}" th:object="${aluno}" method="post">
  <input th:field="*{nome}">
  <input th:field="*{cpf}">
  <input th:field="*{curso}">
  <input th:field="*{email}">
  <input th:field="*{ra}" type="hidden">
</form>

Significado tecnico:

  • th:object="${aluno}" define contexto de bind do model.
  • th:field faz binding automatico para propriedades do DTO/entidade.
  • ra esta oculto no front, mas continua sendo enviado no POST.
  • Script da pagina sincroniza RA com o CPF numerico para atender validacao atual de backend.

8) Relacao com controller atual

Controller principal do fluxo de telas: AlunoController.

  • GET /alunos/novo -> injeta new Aluno() no model e retorna cadastro_aluno.
  • POST /alunos/salvar -> recebe @ModelAttribute Aluno, valida RA, persiste e redireciona.
  • GET /alunos/buscar -> recebe parametro cpf.

9) Contrato front-back (resumo objetivo)

Home

Input enviado: cpf

Endpoint: GET /alunos/buscar

Cadastro

Inputs enviados: nome, cpf, curso, email, ra

Endpoint: POST /alunos/salvar

Render de template

Requisito: model conter atributo aluno

Template: cadastro_aluno.html

10) Boas praticas aplicadas no front

  • Separacao de estilo em CSS dedicado.
  • Hierarquia visual de botoes (primario, secundario, perigo).
  • Estados de foco visiveis (acessibilidade).
  • Campos com placeholders e mascara de CPF.
  • Layout responsivo basico para telas menores.
Resultado: o front esta pronto para o fluxo atual de cadastro/busca e com contrato claro para evolucao do backend.

11) Como apresentar isso para o time em 1 minuto

  1. Mostrar as duas telas e dizer qual e estatica e qual e Thymeleaf.
  2. Explicar que formulario da home chama /alunos/buscar.
  3. Explicar que formulario de cadastro faz bind em Aluno e envia para /alunos/salvar.
  4. Mostrar que RA e oculto no front e preenchido via script com base no CPF.
  5. Concluir com o contrato front-back (campos e endpoints).
